# Oracle–Child Lineage Model（OCLM）

## 1. OCLMとは何か

Oracle–Child Lineage Model（OCLM）は、記録を不変にするためだけの理論ではない。

OCLMが扱うのは、ある存在が、

- どの起源から
- どの親を通じて
- どの時間に
- どの空間で
- どの連続した生成過程によって

生じたのかという、**正統な系譜**である。

記録、署名、データベース、ブロックチェーン、合意結果は、起きた事実を表現し、保存し、検証するための手段である。

しかし、それらの記録を書き換えたとしても、すでに起きた出来事そのものが、別の出来事へ変化するわけではない。

OCLMはこの区別を、形式的な構造として扱う。

> 記録を変更することと、真実の系譜を変更することは同じではない。

## 2. Childの同一性

OCLMにおけるChildは、単独の名前や識別番号によって定義されない。

Childの正統な同一性は、少なくとも次の構成要素の結合によって成立する。

- **Oracle / Origin**  
  起源または生成を認証する存在

- **Parents**  
  Childの生成に関与した親、前身、入力、所有者、状態または系譜

- **Time**  
  生成、承認、継承が成立した時間的条件

- **Space**  
  生成、存在、観測または承認が成立した空間的条件

- **Context**  
  生成に必要となった環境、規則、相互作用および条件

- **Continuity**  
  前の状態から次の状態までが正統に継承されたこと

これらの構成からlineageが導かれ、そのlineageからChildの同一性が導かれる。

概念的には次のように表せる。

```text
Child = G(Oracle, Parents, Time, Space, Context, Continuity)
```

同じ名前、同じ外観、同じID、同じ資産表現を与えたとしても、構成された系譜が異なるものは、元のChildと同一ではない。

それは元のChildを置き換えた存在ではなく、**別の系譜から生じた別の存在**である。

## 3. 正統系譜の非置換性

OCLMの中心命題は、正統系譜の非置換性である。

攻撃者が過去の記録を変更し、異なる親、異なる起源、異なる時間、異なる空間を持つ履歴を作成したとしても、その履歴が元のChildの正統な過去になるわけではない。

偽の記録を作成することは可能である。  
偽の主張を多数へ認めさせることも可能である。  
侵害した鍵によって新たな署名を生成することもあり得る。

しかし、それらによって、すでに成立した生成関係そのものが別の生成関係へ変化することはない。

OCLMが区別するのは次の二つである。

```text
記録上、元の系譜として扱われているもの
```

と、

```text
実際に元の起源から連続して成立した系譜
```

である。

OCLMの目的は、前者が改変された場合でも、後者との違いを判定可能にすることである。

## 4. 記録は真実そのものではない

ブロックチェーン、データベース、署名、証明書、監査ログは重要である。

しかし、それらは真実そのものではなく、真実について提出された証拠である。

多数決は、どの記録を採用するか決められる。  
計算量は、どの履歴を優勢とするか決められる。  
署名は、特定の鍵がデータへ署名したことを示せる。  
台帳は、ネットワーク内部で合意された順序を保存できる。

しかし、それだけでは次の問いに完全には答えられない。

> その記録は、本当に元の存在と同じ起源および連続性を持つのか。

OCLMは、合意された記録と正統な系譜を分離する。

合意が成立したから真実になるのではない。  
正統な系譜を持つものについて、合意や記録が証拠を提供する。

## 5. 複数のブロックチェーンを束ねる上位層

OCLMは、ブロックチェーンを廃棄するための理論ではない。

ブロックチェーンには、すでに膨大な資産、記録、計算能力、合意機構および利用者が存在する。

OCLMは、それらを残したまま、複数のブロックチェーンや台帳を**正統な系譜によって上位から束ねる**。

```text
Blockchain A ─┐
Blockchain B ─┼── OCLM Canonical Lineage
Blockchain C ─┘
```

各ブロックチェーンは、それぞれの内部で成立した記録を提示する。

OCLMは、それらの記録が、

- 同じOriginを持つのか
- 同じChildを指しているのか
- 正統な移転または継承を経ているのか
- 途中で異なる系譜へ置換されていないか
- 分岐した記録のどれが元の系譜を継承しているか

を検証するための上位構造を与える。

ある資産がBlockchain AからBlockchain Bへ移動した場合、Blockchain B上に同じ名前やIDを作成するだけでは、正統な移転にはならない。

Blockchain A上の起源からBlockchain B上のChildまで、切断されていないlineageが必要になる。

したがってOCLMが束ねるのは、単なる台帳データではない。

> 複数のブロックチェーンに分散した記録を、一つの正統な系譜として束ねる。

これがOCLMの上位層としての役割である。

## 6. 鍵侵害と真実

鍵が侵害された場合、攻撃者は偽の署名や証明書を生成できる可能性がある。

しかし、鍵侵害によって過去に起きた生成事実そのものが変化するわけではない。

侵害が変えるのは、真実ではなく、**その鍵を根拠とする証拠の信頼性**である。

OCLMでは、どの鍵が、いつ、どのepochで侵害されたかを系譜と分離して扱う。

後のepochで鍵が侵害されても、すでに消去された過去epochの鍵や、それ以前に正当に成立したChildの系譜が自動的に書き換わるわけではない。

侵害後に生成された偽の記録は、侵害後の証拠として識別されるべきであり、侵害以前の正統な生成事実と同一視されてはならない。

## 7. Phase 2で検証したこと

OCLM Phase 2では、Phase 1の形式モデルを破壊することを目的として、80種類の象徴的な敵対攻撃クラスを構築した。

攻撃対象には次が含まれる。

- 同一Childの異なる系譜への置換
- Origin、Parents、Time、Spaceの差し替え
- 過去epochの偽造
- 後続鍵侵害による過去の書き換え
- credentialおよびcertificateの偽造
- 異なる系譜証拠の継ぎはぎ
- rollback
- revocation後の復活
- split-view
- conflicting transparency roots
- 同一provenanceからの異なるChildの正統化

同時に、モデルが何も実行できないため攻撃も成立しないという状態を排除するため、8種類の健全性・到達可能性命題を検証した。

最終結果は次のとおりである。

```text
Symbolic adversarial attack classes: 80
Attack classes ruled out: 80/80
Counterexample traces found: 0
Unresolved attack classes: 0

Health and reachability properties: 8
Reachable properties: 8/8
```

最後まで未決着だったA48については、次の三つの補助命題へ分解した。

```text
H1:
Same Child ID
→ Same lineage

H2:
Same generated lineage
→ Same public ancestry tuple

H3:
Uncompromised acceptance
→ Matching generated origin
```

その上で、H1・H2・H3と最終A48命題を同一のTamarin実行内で検証した。

結果として、対象epoch鍵またはRoot鍵の侵害を伴わずに、同一Childの過去を異なる系譜へ置換する反例は、定義されたモデルの範囲内では成立しなかった。

## 8. 検証結果の意味

Phase 2の結果は、現実世界に存在し得るすべての攻撃を列挙したという意味ではない。

また、あらゆる実装、Oracle、センサー、ネットワーク、運用者が自動的に安全になることを意味しない。

この結果が示すのは、次の範囲である。

- 定義されたOCLM形式モデル
- モデル化された暗号プリミティブ
- 明示された攻撃者能力
- 定義された鍵侵害条件
- 80種類の象徴的攻撃クラス
- 8種類の健全性・到達可能性条件

この範囲において、OCLMの中心命題である正統系譜の非置換性を破る反例は確認されなかった。

現実の実装では、Oracleの信頼境界、物理世界との接続、鍵管理、時刻証明、空間証明、運用上の権限分離が別途必要になる。

## 9. OCLMの位置付け

OCLMは、ブロックチェーンそのものではない。  
新しい多数決方式でもない。  
新しい最長鎖ルールでもない。  
単独の署名方式でもない。

OCLMは、それらによって生成された証拠を、起源、親、時間、空間および連続性の観点から結合するための理論である。

> ブロックチェーンが記録を保持し、OCLMがその記録の系譜を束ねる。

個々のチェーンが内部の状態を確定し、OCLMがチェーンを横断する正統な系譜を確定する。

OCLMが目指すのは、壊れない記録を一つ作ることだけではない。

> 記録が壊れ、分岐し、移動し、置換を主張されたとしても、元の存在がどの系譜から生じたのかを失わない構造を作ることである。
